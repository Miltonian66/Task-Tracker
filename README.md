# Task Tracker — Внутренний сервис учёта задач

Минимальный сервис для отслеживания задач.

## Стек

- Python 3.11+
- FastAPI
- SQLAlchemy
- PostgreSQL 16
- Redis 7
- Docker Compose
- Pytest + Ruff

---

## Запуск

```bash
docker compose up --build
```

Когда поднимутся все три сервиса (`db`, `redis`, `api`), API доступно на:

- Swagger UI: <http://localhost:8000/docs>
- ReDoc: <http://localhost:8000/redoc>
- Health: <http://localhost:8000/health>

## API

| Метод     | Путь                       | Что делает                                                              |
| --------- | -------------------------- |-------------------------------------------------------------------------|
| `POST`    | `/tasks`                   | Создать задачу (стартует в `todo`)                                      |
| `GET`     | `/tasks`                   | Список задач; фильтры `status`, `assignee`; пагинация `limit`, `offset` |
| `GET`     | `/tasks/{id}`              | Получить задачу                                                         |
| `PATCH`   | `/tasks/{id}/status`       | Сменить статус (строго на следующий шаг)                                |
| `DELETE`  | `/tasks/{id}`              | Удалить задачу                                                          |
| `GET`     | `/health`                  | Readiness                                                               |

**Пагинация:** `limit` - целое 1..500 включительно (по-умолчанию 100), `offset` - >= 0.

## Пример

```bash
# Создать задачу
curl -sX POST http://localhost:8000/tasks \
  -H 'Content-Type: application/json' \
  -d '{"title":"Название","description":"Описание","assignee":"milton66"}'

# Список задач для исполнителя в статусе in_progress
curl -s 'http://localhost:8000/tasks?status=in_progress&assignee=milton66'

# Перевести в работу
curl -sX PATCH http://localhost:8000/tasks/<UUID>/status \
  -H 'Content-Type: application/json' \
  -d '{"status":"in_progress"}'
```

## Формат ошибок

Все ошибки приходят в едином виде:

```json
{
  "error": {
    "code": "invalid_status_transition",
    "message": "Прыжки через статусы запрещены: 'todo' -> 'done'. Ожидался следующий шаг: 'todo' -> 'in_progress'.",
    "details": { "current": "todo", "requested": "done", "expected": "in_progress" }
  }
}
```

| `code`                       | HTTP | Когда                                                              |
| ---------------------------- | ---- | ------------------------------------------------------------------ |
| `validation_error`           | 422  | Pydantic не пропустил входные данные                               |
| `task_not_found`             | 404  | Задача с таким `id` не найдена                                     |
| `invalid_status_transition`  | 409  | Откат назад / прыжок через шаг / выход из `done`                   |
| `database_error`             | 500  | Непредвиденная ошибка БД. В `details.request_id` — токен для логов; внутренний тип/трейс наружу не уходит |

## Схема

| Поле          | Тип        | Обязательное | Комментарий                                                                                         |
| ------------- | ---------- | ------------ |-----------------------------------------------------------------------------------------------------|
| `id`          | `uuid`     | системное    | UUID, а не auto-increment: безопаснее, даёт меньше информации о данных в БД                         |
| `title`       | `str(200)` | да           | Короткое название                                                                                   |
| `description` | `str(1..2000)` | да       | Описание задачи                                                                                     |
| `assignee`    | `str(100)` | да           | Логин/имя исполнителя. Строка, а не id user'а, т.к. требуется MVP                                   |
| `status`      | `enum`     | системное    | Стартует в `todo`, меняется через `PATCH /tasks/{id}/status`                                        |
| `created_at`  | `datetime` | системное    | Дополнительное поле. Очень важная информация о задаче. Так команда сможет увидеть "зависшие" задачи |
| `updated_at`  | `datetime` | системное    | Дополнительное поле. Для анализа последнего изменения задачи (debug)                                |

## Статусы

```
todo  ──►  in_progress  ──►  review  ──►  done
```

1. Разрешён только переход на один статус вперёд.
2. Запрещены переходы на ранние статусы в цепочке.
3. Запрещены скачки через статусы (`todo` -> `done` или `in_progress` -> `done`).
4. Запрещён переход на тот же статус (сигнализирует об ошибке).

## TTL

Выбран равным 30 секундам.

Причины:

1. Меньше храним мусорных ключей от прошлых версий.  
2. Возвращаем достаточно актуальные данные.
3. Достаточно, чтобы при burst чтении снять нагрузку с сервиса.
